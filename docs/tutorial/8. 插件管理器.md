# 8. 插件管理器

## 8.1 简介

插件管理器（Plugin Manager）是一个功能完备的插件管理单例类，负责扫描、安装、加载、卸载和启用/禁用插件，支持依赖解析、完整性检查、核心/第三方插件区分、多种安装方式（复制/链接）以及 ZIP 包安装。它使用 JSON 清单管理插件元数据，通过动态库加载实现插件的运行时扩展。

## 8.2 特性

* **双清单管理**：核心插件清单（core_manifest.json）和用户插件清单（user_manifest.json）分离；
* **依赖解析**：自动递归加载插件依赖，检测循环依赖；
* **完整性检查**：验证插件目录、二进制文件、资源文件是否存在，ID 是否匹配；
* **多种安装方式**：支持两种安装方式，分别是 Copy 方式和 Link 方式， Copy 方式将插件文件复制到用户目录而 Link 方式直接引用外部路径（如开发或使用大型插件时）；
* **插件打包**：支持从 ZIP 包安装插件；
* **状态控制**：启用/禁用插件，禁用时级联禁用所有依赖它的插件；
* **懒加载**：仅在需要时加载 DLL，支持批量加载；

## 8.3 核心概念

### 8.3.1 插件类型

插件类型（PluginType）分为两种，分别是 Builtin （核心插件，位于核心目录，不可禁用）和 ThirdParty （第三方插件，位于用户目录，可启用/禁用）。头文件中的定义如下：

```cpp
// lgcPluginManager.hpp
// 插件类型
enum class PluginType
{
    Builtin,
    ThirdParty
};
```

### 8.3.2 安装方式

安装方式（InstallMethod）分为两种，分别是 Copy （插件文件复制到用户目录，卸载时删除）和 Link （直接引用外部路径，卸载时仅移除引用，保留源文件）。头文件中的定义如下：

```cpp
// lgcPluginManager.hpp
// 安装方式
enum class InstallMethod
{
    Copy,
    Link
};
```

### 8.3.3 插件状态

插件状态（PluginState）， 分为 Unloaded 、 Loaded 、 Disabled 、 Broken 和 Error 。具体含义请参考代码注释：

```cpp
// lgcPluginManager.hpp
// 运行状态
enum class PluginState
{
    Unloaded,   // 未加载（在清单中，但未加载DLL）
    Loaded,     // 已加载（DLL loaded & Init success）
    Disabled,   // 被禁用
    Broken,     // 文件损坏或缺失
    Error       // 加载过程出错
};
```

### 8.3.4 插件元数据 (PluginMetadata)

请参考下文的注释：

```cpp
// lgcPluginManager.hpp
// 插件元数据
struct PluginMetadata
{
    std::string id; // 唯一标识符
    std::string name; // 显示名称
    std::string version; // 版本号
    std::string binaryName; // 主二进制文件名（如 plugin.dll）

    std::string defaultValuePath; // 插件默认配置路径（可选）

    std::vector<std::string> libraries; // 依赖的附加 DLL 列表（如 lib1.dll）
    std::vector<std::string> resources; // 资源文件列表（如图片、配置文件）
    std::vector<std::string> dependencies; // 依赖的其他插件 ID 列表

    std::filesystem::path rootPath; // 插件根目录
    PluginType type = PluginType::ThirdParty; // 插件类型
    InstallMethod installMethod = InstallMethod::Copy; // 安装方式

    bool isEnabled = true; // 是否启用
    std::string integrityStatus = "Unknown"; // 完整性检查结果
};

```

### 8.3.5 插件实例

插件实例（PluginInstance），包含元数据、当前状态、DLL 句柄、插件对象指针。
```cpp
// lgcPluginManager.hpp
// 插件实例
class PluginInstance
{
public:
    PluginMetadata metadata;
    PluginState state = PluginState::Unloaded;
    void* dllHandle = nullptr;

    std::shared_ptr<PluginBase> pluginObject;

};
```

### 8.3.6 插件基类

插件基类（PluginBase），这是所有插件必须实现的抽象类，定义在 virPluginBase.hpp 中：

```cpp
// virPluginBase.hpp
#pragma once
#include "defCoreApi.hpp"
#include <string>
#include <atomic>

// 纯虚接口
namespace core
{
	class PluginBase
	{
	public:
		PluginBase() = default;
		virtual ~PluginBase() = default;

		// 获取插件信息
		virtual const std::string getId() const = 0;
		virtual const std::string getName() const = 0;
		virtual const std::string getVersion() const = 0;
		virtual const std::string getCert() const = 0;

		// 【关键】安装插件：传入主窗口指针，允许插件添加菜单、工具栏等
		virtual bool init() = 0;

		// 【关键】卸载插件：清理添加的菜单项、释放资源
		virtual void shutdown() = 0;

	};
}
```

注意：插件需要导出两个 C 风格函数：
```cpp
extern "C" PluginBase* createPlugin();

extern "C" void destroyPlugin(PluginBase*);
```

以下是一个插件实例的例子：

```cpp
#pragma once
#include "defMyPluginApi.hpp"
#include <Plugin/virPluginBase.hpp>

// 强制检查：如果插件和主程序 CRT 设置不一致，编译报错
#if defined(_MT) && !defined(_DLL)
#error "Must use Dynamic CRT (/MD or /MDd)"
#endif

namespace ess
{
	class MyPlugin : public core::PluginBase
	{
	public:
		MyPlugin() {}
		virtual ~MyPlugin() = default;

		// 获取插件信息
		const std::string getId() const override { return "com.my_company.my_plugin"; }
		const std::string getName() const override { return "my_plugin"; }
		const std::string getVersion() const override { return "1.0.0"; }
		const std::string getCert() const override { return ""; }

		bool init() override
		{
            // TODO
			return true;
		}

		void shutdown() override
		{
			// TODO
		}

	};
}

extern "C"
{
	MYPLUGIN_API core::PluginBase* createPlugin();

	MYPLUGIN_API void destroyPlugin(core::PluginBase* plugin);
}
```

## 8.4 使用方法

### 8.4.1 初始化

在程序启动时，lgcCore.cpp 中的 initializeCore() 会自行调用 Plugin Manager 的 init() 方法，往往插件管理器应该被最后调用并最早被释放：

```cpp
// lgcCore.cpp
void initializeCore(
	const std::string& appName,
	const std::string& confPath,
	const std::string& coreConfPath,
	const std::string& corePluginsDir,
	const std::string& userPluginsDir
)
{
    // ......
	PluginManager::get().init(corePluginsDir, userPluginsDir);
}
```

由于几个管理器之间有依赖关系，初始化的顺序不能被随意改变，因此**在组合使用时不推荐自行调用 init()** （除非你真的非常了解你在做什么！）。

当然如果想要单独拿出来用也是可以的，记得去掉其中的 Log Manager 避免出现编译错误。此时自行调用 init() 进行初始化。

```cpp
// 获取单例
core::PluginManager& pm = core::PluginManager::get();
// 初始化
std::string corePluginsDir, userPluginsDir;
bool isSucceeded = pm.init(corePluginsDir, userPluginsDir);
```

### 8.4.2 插件安装

使用 installFromPath() 从路径安装（即 Link 方式），插件元数据将会写入用户清单而不复制插件，会自动执行完整性检查并尝试加载。

```cpp
// 假设已经有一个安装到某个位置的插件
std::string externalPath; // externalPath 是包含 plugin.json 的插件目录
bool isSucceeded = pm.installFromPath(externalPath);
```

使用 installFromPackage() 从 ZIP 包安装（即 Copy 方式），这将会使 ZIP 解压到 userDir/{id} 目录，如果插件已经存在，将覆盖已存在的第三方插件（不能覆盖核心插件），并且将自动注册并尝试加载。

```cpp
// 假设已经有一个压缩包
std::string zipPath; // zipPath 是 zip 目录
bool isSucceeded = pm.installFromPackage(zipPath);
```

### 8.4.3 插件卸载

使用 uninstallPlugin() 卸载插件，仅能卸载第三方插件，如果插件已加载，将先卸载 DLL ，如果安装方式为 Copy ，将删除插件目录，随后无论是 Copy 还是 Link ，都将从用户清单中移除条目。

```cpp
// 假设已经知道了准备卸载的插件的 ID
std::string id;
bool isSucceeded = pm.uninstallPlugin(id);
```

### 8.4.4 插件加载

Plugin Manager 采用懒加载策略，插件在第一次请求时加载（或调用 loadPlugin()）。

使用 loadPlugin() 加载单个插件，这将自动递归加载依赖，若插件已加载或禁用/损坏，返回 false ，若加载成功将插件状态置为 Loaded ，并加入加载顺序列表。

```cpp
// 假设已经知道了准备加载的插件的 ID
std::string id;
bool isSucceeded = pm.loadPlugin(id);
```

使用 loadAll() 加载所有插件，这将遍历所有已启用且未加载的插件，并依次调用 loadPlugin() 。

```cpp
pm.loadAll();
```

### 8.4.5 启用/禁用插件

使用 setPluginEnabled() 启用与禁用插件，这将返回受影响的插件 ID 列表（包括级联禁用的插件）。将插件标记为启用时，状态将会被置为 Unloaded，并尝试加载；将插件标记为禁用时，将会递归禁用所有依赖此插件的插件，并卸载当前插件（如果已加载）。

```cpp
std::string pluginId = "some.plugin";
auto affected = pm.setPluginEnabled(pluginId, false);
APP_LOG_INFO("Disabled plugins: {}", affected.size());
```

### 8.4.6 插件查询

使用 getPlugin() 获取某个插件的实例，使用 getAllPlugins() 获得所有插件的实例，使用 getMissingCorePlugins() 获取缺失的插件 ID 。随后可以通过 PluginInstance 可以访问其 metadata 和 state 。它们具体的函数签名如下：

```cpp
std::shared_ptr<PluginInstance> getPlugin(const std::string& id);
std::vector<std::shared_ptr<PluginInstance>> getAllPlugins() const;
std::vector<std::string> getMissingCorePlugins() const;
```

### 8.4.7 关闭

使用 shutdown() 卸载所有插件，释放资源。析构函数将自动调用 shutdown() 。

```cpp
pm.shutdown();
```

## 8.5 插件开发步骤

1. 实现 PluginBase 派生类并导出 C 风格工厂函数，上文已经给出示例，这里不再赘述；
2. 提供 plugin.json
放在插件根目录，例如：

```json
{
    "id" : "com.my_company.my_plugin",
    "name" : "my_plugin",
    "version" : "1.0.0",
    "binary" : "my_plugin.dll",
    "default_value_path" : "default_value.json"
}
```

注册成功后，会写入到相应的清单，清单文件有两个，分别是 core_manifest.json 和 user_manifest.json，这里给出示例：

核心清单 (core_manifest.json)

```json
{
    "plugins": {
        "com.my_company.my_plugin": {
            "path": "./my_plugin",
            "method": "Copy", 
            "version": "1.0.0"
        }
    }
}
```

用户清单 (user_manifest.json)

```json
{
    "plugins": {
        "my.plugin": {
            "path": "C:/Users/.../myplugin",
            "method": "Link",
            "enabled": true
        },
        "other.plugin": {
            "path": "./other_plugin",
            "method": "Copy",
            "enabled": false
        }
    }
}
```

## 8.6 完整示例

```cpp
#include "Plugin/lgcPluginManager.hpp"
#include "Log/lgcLogManager.hpp"

int main()
{
    core::LogManager::init();

    auto& pm = core::PluginManager::get();
    if (!pm.init("./core", "./user"))
    {
        APP_LOG_ERROR("Failed to init PluginManager");
        return 1;
    }

    auto missing = pm.getMissingCorePlugins();
    if (!missing.empty())
    {
        for (const auto& m : missing)
            APP_LOG_ERROR("Missing core plugin: {}", m);
        return 1;
    }

    // 列出所有插件
    for (auto& plugin : pm.getAllPlugins())
    {
        APP_LOG_INFO("Plugin: {} [{}] - State: {}",
                     plugin->metadata.id,
                     plugin->metadata.version,
                     static_cast<int>(plugin->state));
    }

    // 加载所有插件
    pm.loadAll();

    // 获取插件实例并调用
    auto plugin = pm.getPlugin("my.plugin");
    if (plugin && plugin->pluginObject)
    {
        // 可以通过 plugin->pluginObject 调用插件方法
    }

    // 禁用插件
    pm.setPluginEnabled("some.plugin", false);

    // 安装新插件
    pm.installFromPackage("plugins/new_plugin.zip");

    // 程序退出时自动 shutdown
    return 0;
}
```

## 8.7 注意事项

* 线程安全： Plugin Manager 内部未使用锁，非线程安全。所有操作应在主线程或单一线程中执行。若需多线程访问，请外部加锁；
* 循环依赖检测：加载插件时会检测循环依赖，若发现循环依赖，加载失败并记录错误；
* 损坏处理：当核心插件损坏时，将记录到 mMissingCorePlugins ，程序应据此终止；当采用 Copy 方式的第三方插件损坏时，自动从清单移除；当采用 Link 方式的第三方插件损坏时，保留条目，状态设为 Broken ；
* DLL 加载路径：Windows 上使用 LOAD_WITH_ALTERED_SEARCH_PATH 标志，确保 DLL 能加载同目录下的依赖库；
* 导出函数名称修饰：工厂函数必须使用 extern "C" 防止名称修饰，并确保函数名正确（createPlugin/destroyPlugin）；
* 卸载顺序：插件按加载顺序反向卸载，确保依赖先于被依赖卸载；
* 日志：必须在使用前确保日志管理器（Log Manager）已经初始化，或注释掉相关的日志函数；
* 跨平台注意事项：Linux 需使用 -fvisibility=hidden 控制符号可见性，并显式导出工厂函数；路径处理使用 std::filesystem::path，注意跨平台分隔符。
