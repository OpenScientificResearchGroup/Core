# 2. 命令管理器

## 2.1 简介

命令管理器（Command Manager）是一个基于命令模式（Command Pattern）的撤销/重做管理器，支持命令的提交、撤销、重做、事务合并、脏状态跟踪等功能。它使用 std::deque 存储命令历史，并通过索引管理当前状态。

## 2.2 特性
* **命令模式**：每个操作封装为 CommandBase 派生类，实现 execute / undo / redo；
* **撤销/重做**：支持撤销/重做（受限于历史队列大小）；
* **命令合并**：相邻同类命令可合并（如连续移动操作合并为一次）；
* **事务支持**：将多个命令打包为一个原子操作（使用 CompositeCommand）；
* **脏状态跟踪**：记录文档是否已保存，便于标题栏显示 *；
* **历史大小限制**：自动裁剪超出最大历史条目的命令；
* **线程安全**：考虑到在软件中在同一时刻只能执行一个命令，因此类本身非线程安全，若跨线程使用需外部加锁。

## 2.3 必须的基类

命令管理器需要 CommandBase 类，任何想要保存在命令管理器中的命令必须继承这个类并实现相关的函数。

```cpp
// virCommandBase.hpp
#pragma once
#include "defCoreApi.hpp"

#include <memory>
#include <string>

namespace core
{
	class CommandBase
	{
	public:
		CommandBase() = default;
		virtual ~CommandBase() = default;

		CommandBase(const CommandBase&) = delete; // 基类禁用拷贝，防止切片问题
		CommandBase& operator=(const CommandBase&) = delete;
		CommandBase(CommandBase&&) = default; // 默认构造和移动通常需要保留
		CommandBase& operator=(CommandBase&&) = default;

		// --- 1. 核心逻辑 ---
		// 返回 bool：因为执行可能会失败（例如磁盘满了、内存不够、文件被占用）
		// 如果返回 false，命令历史栈不应记录此命令
		virtual bool execute() = 0;

		virtual void undo() = 0;

		// 通常 Redo 和 Execute 逻辑是一样的，但也可能有细微区别
		// 默认实现可以是直接调用 Execute()
		virtual void redo() { execute(); }

		// --- 2. UI 交互支持 ---
		// 获取命令名称，用于在菜单中显示。
		// 例如：菜单显示 "撤销 移动对象" (Undo Move Object)
		virtual std::string getName() const = 0;

		// --- 3. 性能优化 (命令合并) ---
		// 用于将连续的微小操作合并为一个。
		// 例如：用户拖动滑块调整亮度，产生了 50 个命令，
		// 我们希望这 50 个命令合并成 1 个，否则用户要按 50 次 Ctrl+Z。
		// 【修改说明】：这里不能使用 std::unique_ptr<CommandBase> (按值传递)，
		// 也不建议使用 const std::unique_ptr<CommandBase>& (引用传递)。
		// 使用 const CommandBase* (裸指针) 是最佳实践。
		virtual bool mergeWith(const CommandBase* other) { return false; }

		// --- 4. 识别与分类 ---
		// 获取命令的唯一 ID 或类型。
		// 用于判断两个命令是否属于同一种类，辅助 MergeWith 判断。
		virtual std::string getId() const = 0;

		// --- 5. 内存管理 (可选，针对大型数据) ---
		// 返回该命令占用的内存大小（估算）。
		// 如果历史栈占用内存过大（如 500MB），管理器可以根据这个值清理最早的命令。
		virtual size_t getSizeInBytes() const { return sizeof(*this); }

		// --- 6. 控制是否入栈 ---
		// 有些命令不适合入栈（如保存命令），可以重载此方法返回 false
		virtual bool allowStack() const { return true; }

	};
}// namespace core
```

为了支持事务，命令管理器也需要 CompositeCommand 类。

```cpp
// lgcCompositeCommand.hpp
#pragma once
#include "defCoreApi.hpp"
#include "virCommandBase.hpp"

#include <vector>
#include <memory>
#include <string>

namespace core
{
	class CORE_API CompositeCommand : public CommandBase
	{
	public:
		CompositeCommand(const std::string& name = "Composite", const std::string& id = "/Core/composite_command");
		virtual ~CompositeCommand() = default;

		CompositeCommand(const CompositeCommand&) = delete; // 禁止拷贝构造 (因为 unique_ptr 不能拷贝)
		CompositeCommand& operator=(const CompositeCommand&) = delete; // 禁止拷贝赋值 (防止编译器生成导致报错的代码)
		CompositeCommand(CompositeCommand&&) = default; // 允许移动构造 (将所有权转移，这是 unique_ptr 支持的)
		CompositeCommand& operator=(CompositeCommand&&) = default;// 允许移动赋值

		bool execute() override;
		void undo() override;
		void redo() override;
		std::string getName() const override;
		std::string getId() const override;

		void append(std::unique_ptr<CommandBase> cmd);
		bool isEmpty() const;
		void setName(const std::string& name);
		void setId(const std::string& id);

	private:
		std::vector<std::unique_ptr<CommandBase>> mCmds;
		std::string mName, mId;

	};
} // namespace core
```

这两个头文件和 lgcCommandManager.hpp 在同一个文件夹中。

## 2.4 使用方法

### 2.4.1 初始化

由于命令管理器和文件强相关，故设计时是作为文件类的成员存在，不设置单例，初始化时需要手动进行初始化。可以通过构造函数或 init() 初始化：

```cpp
// 设置历史记录最多保存为 50 条
size_t maxHistory = 50;
core::CommandManager mCommandManager(maxHistory);
// 或者
// mCommandManager.init(maxHistory);
```

可在构造后重新设置最大历史长度，返回 true 表示成功。

### 2.4.2 提交

先创建一个命令类的智能指针，随后转移到 Command Manager 中。

```cpp
// 假设这是一个新建文件的命令，继承自 CommandBase 并且已经实现相应接口
std::unique_ptr<New> cmdPtr = std::make_unique<New>();
mCommandManager.submit(std::move(cmdPtr));
```

### 2.4.3 撤销与重做

```cpp
// 撤销命令
mCommandManager.undo();
// 重做命令
mCommandManager.redo();
```

### 2.4.4 清空历史

清空所有历史，重置索引和保存点。

```cpp
mCommandManager.clear();
```

### 2.4.5 事务开始与结束

调用后 beginTransaction() ，后续的 submit 不会直接进入主历史，而是暂存到内部 mPendingTransaction，并且直到调用 endTransaction() 才作为一个命令提交到命令管理器中。

```cpp
// 给这个事务取个名~
mCommandManager.beginTransaction("画矩形");
mCommandManager.submit(std::make_unique<SetColorCmd>("red"));
mCommandManager.submit(std::make_unique<DrawRectCmd>(10, 10, 100, 50));
mCommandManager.endTransaction();
```

此时如果用户执行一次撤销，会同时撤销 SetColor 和 DrawRect。

### 2.4.6 脏状态（保存点）

这部分主要用来跟踪文档是否已保存，便于界面显示 * 未保存标记。 markAsSaved() 通常在用户执行“保存”操作时调用，将当前索引记录为保存点。 isDirty() 则用来查询查询是否为脏（是否被修改了）。

```cpp
// 标记为已保存
mCommandManager.markAsSaved();
// 查询是否脏
bool isDirty = mCommandManager.isDirty();
```

* 若由于历史裁剪导致保存点被移除，isDirty() 将在下一次保存前始终返回 true；
* 初始状态为未保存（除非手动调用 markAsSaved() ）。

### 2.4.8 状态查询

能否撤销/重做，允许将返回 true ，不允许将返回 false 。

```cpp
// 查询是否能撤销
bool isUndoAllowed = mCommandManager.allowUndo();
// 查询是否能重做
bool isRedoAllowed = mCommandManager.allowRedo();
```

当然，为了 Ui 能更加智能，还支持获取撤销/重做命令名称。

```cpp
std::string undoName = mCommandManager.getUndoName();  // 例如 "撤销 移动"
std::string redoName = mCommandManager.getRedoName();  // 例如 "重做 移动"
```

若不可撤销/重做，返回空字符串。

### 2.4.9 清理与关闭

调用 clear() 将清空命令管理器中的全部成员变量。 shutdown() 本质上也是在调用 clear() ，只是为了统一一下管理器的接口就加了一个。

```cpp
mCommandManager.clear()；
mCommandManager.shutdown();
```

在析构函数中也会调用 shutdown() 。

## 2.5 完整示例
假设我们有一个简单的绘图程序，实现了 MoveCommand（移动图形）和 ColorCommand（修改颜色）。

```cpp
#include "Command/lgcCommandManager.hpp"
#include <iostream>

// 自定义命令示例
class MoveCommand : public core::CommandBase
{
public:
    MoveCommand(int dx, int dy)
        : CommandBase(), mDx(dx), mDy(dy)
    {

    }
    
    bool execute() override
    {
        std::cout << "执行移动: (" << mDx << "," << mDy << ")\n";
        // 实际移动逻辑...
        return true;
    }

    void undo() override
    {
        std::cout << "撤销移动: (" << -mDx << "," << -mDy << ")\n";
    }

    void redo() override
    {
        execute();
    }

    std::string getName() const override
    {
        return "移动";
    }

    bool mergeWith(core::CommandBase* other) override
    {
        // 合并连续的移动命令
        if (auto* move = dynamic_cast<MoveCommand*>(other))
        {
            mDx += move->mDx;
            mDy += move->mDy;
            return true; // 合并成功，丢弃 other
        }
        return false;
    }

    std::string getId() const override
    {
        return "/command/move";
    }

private:
    int mDx, mDy;

};

int main()
{
    core::CommandManager mgr(10); // 最多保留10步历史

    // 提交几个命令
    mgr.submit(std::make_unique<MoveCommand>(5, 0));
    mgr.submit(std::make_unique<MoveCommand>(3, 2)); // 应该与上一个合并
    mgr.submit(std::make_unique<MoveCommand>(1, 1)); // 合并后最终偏移 (9,3)

    std::cout << "当前可撤销: " << mgr.getUndoName() << "\n"; // 输出"移动"

    // 撤销
    mgr.undo(); // 输出"撤销移动: (-9,-3)"

    // 重做
    mgr.redo(); // 输出"执行移动: (9,3)"

    // 事务示例
    mgr.beginTransaction("批量操作");
    mgr.submit(std::make_unique<MoveCommand>(2, 2));
    mgr.submit(std::make_unique<MoveCommand>(3, 3));
    mgr.endTransaction(); // 这两个移动被合并为一个复合命令

    mgr.undo(); // 撤销整个事务

    // 脏状态
    mgr.markAsSaved();
    std::cout << "是否脏: " << mgr.isDirty() << "\n"; // 0 (false)
    mgr.submit(std::make_unique<MoveCommand>(1, 0));
    std::cout << "是否脏: " << mgr.isDirty() << "\n"; // 1 (true)

    return 0;
}
```

## 2.6 注意事项

* 日志：必须在使用前确保日志管理器（Log Manager）已经初始化，或注释掉相关的日志函数；
* 命令所有权：submit() 接受 std::unique_ptr\<CommandBase\>，命令管理器将获得所有权，确保命令在历史中存活；
* 合并条件：只有相邻且类型相同的命令才能合并，并且需在命令类中的 mergeWith() 中实现具体逻辑；
* 事务限制：不支持嵌套事务，连续调用 beginTransaction() 会导致错误日志，应确保成对调用；
* 线程安全：CommandManager 本身非线程安全，若在多线程环境中使用需自行加锁；
* 性能：历史队列使用 std::deque，头部删除效率高；但合并逻辑可能遍历命令，建议命令不要过于庞大。
