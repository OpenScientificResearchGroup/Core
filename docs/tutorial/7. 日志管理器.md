# 7. 日志管理器

## 7.1 简介

日志管理器（Log Manager）是一个基于 spdlog 的日志管理单例类，提供异步日志记录、文件轮转、控制台输出（Debug 模式自动分配控制台）、VS 输出窗口支持、崩溃前后日志缓冲（backtrace）等功能。通过预定义的宏，可以方便地在代码任何位置记录带文件名和行号的日志。

## 7.2 特性

* **异步日志**：使用 spdlog 异步线程池，避免日志 I/O 阻塞主线程。
* **文件轮转**：支持日志文件大小限制和自动分割（默认 5MB，最多 3 个文件）。
* **多级日志**：TRACE、DEBUG、INFO、WARN、ERROR、CRITICAL 六级。
* **调试增强**：Debug 模式下可分配控制台窗口（Windows），输出彩色日志。同时支持 VS 输出窗口（OutputDebugString）（已注释，可按需开启）。
* **崩溃防护**：启用 backtrace，最近 32 条日志缓存，发生 CRITICAL 错误时自动 dump。定期刷新（每 3 秒），减少崩溃导致日志丢失的风险。
* **线程安全**：使用互斥锁保证单例初始化的安全性，spdlog 自身也是线程安全的。
* **简单宏接口**：使用 APP_LOG_XXX 宏即可输出一条日志。

## 7.3 使用方法

### 7.3.1 初始化

在程序启动时，lgcCore.cpp 中的 initializeCore() 会自行调用 I18n Manager 的 init() 方法：

```cpp
// lgcCore.cpp
void initializeCore(
	const std::string& appName,
	const std::string& confPath,
	const std::string& coreConfPath,
	const std::string& corePluginsDir,
	const std::string& userPluginsDir
)
{
    LogManager::init();
    ConfigManager::get().init(confPath, coreConfPath);
    LogManager::init(
        ConfigManager::get().getValue<std::string>("/Core/log/path"),
        ConfigManager::get().getValue<size_t>("/Core/log/max_file_size"),
        ConfigManager::get().getValue<size_t>("/Core/log/max_files"),
        ConfigManager::get().getValue<bool>("/Core/log/console_out")
    );
    // ......
}
```

由于几个管理器之间有依赖关系，初始化的顺序不能被随意改变，因此**在组合使用时不推荐自行调用 init()** （除非你真的非常了解你在做什么！）。

当然如果想要单独拿出来用也是可以的，此时使用 init() 对 Log Manager 进行初始化：

```cpp
// 最简单的初始化，使用默认参数（无控制台输出）
core::LogManager::init();
// 自定义配置：保存日志到 ./logs/app.log，10MB 轮转，保留 5 个文件，启用控制台
std::string path = "./logs/app.log";
size_t maxFileSize = 10 * 1024 * 1024;
size_t maxFiles = 5;
bool consoleOut = true;
core::LogManager::init(path, maxFileSize, maxFiles, consoleOut);
```

注意：
* init 可以多次调用，每次调用会关闭现有日志系统并重新初始化；
* 在 Release 模式下，即使 consoleOut 为 true，也不会自动分配控制台窗口（仅 Debug 模式有此行为）；
* 如果使用其他的管理器，务必确保 Log Manager 最早初始化，或注释掉相关的日志代码。

### 7.3.2 日志宏

预定义的宏可以直接在代码中使用，输出的日志中将自动包含当前文件名和行号，日志分为六种不同的级别，详情请查看下表：

| 宏 | 对应级别 | 使用示例 |
| --- | --- | --- |
| APP_LOG_TRACE(...) | trace | APP_LOG_TRACE("Entering function with param: {}", x); |
| APP_LOG_DEBUG(...) | debug | APP_LOG_DEBUG("Value: {}", value); |
| APP_LOG_INFO(...) | information | APP_LOG_INFO("Application started."); |
| APP_LOG_WARN(...) | warning | APP_LOG_WARN("Disk space low: {} MB", freeSpace); |
| APP_LOG_ERROR(...) | error | APP_LOG_ERROR("Failed to open file: {}", filename); |
| APP_LOG_CRITICAL(...) | critical | APP_LOG_CRITICAL("Unrecoverable error, terminating."); |

所有宏均支持 fmt 风格格式化（{} 占位符），可传入任意数量参数。

```cpp
int userCount = 42;
APP_LOG_INFO("当前用户数量: {}", userCount);

try
{
    // 可能抛出异常的代码
}
catch (const std::exception& e)
{
    APP_LOG_ERROR("在用户数量为 {} 时，发生异常: {}", userCount, e.what());
}
```

### 7.3.3 清理与关闭

使用 shutdown() 关闭日志，这将刷新所有日志，释放资源，关闭控制台（如果已分配）。

程序退出时通常无需显式调用，因为静态析构会自动调用 shutdown。

```cpp
LogManager::shutdown();
```

## 7.4 完整示例
```cpp
#include "Log/lgcLogManager.hpp"
#include <thread>
#include <vector>

void worker(int id)
{
    for (int i = 0; i < 5; ++i)
    {
        APP_LOG_DEBUG("Worker {} is processing item {}", id, i);
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
    }
    APP_LOG_INFO("Worker {} finished", id);
}

int main()
{
    // 初始化日志系统，启用控制台输出（Debug 模式下会弹出控制台窗口）
    core::LogManager::init("./logs/myapp.log", 2 * 1024 * 1024, 5, true);

    APP_LOG_INFO("Application started");

    // 模拟多线程日志
    std::vector<std::thread> threads;
    for (int i = 0; i < 3; ++i)
        threads.emplace_back(worker, i);

    for (auto& t : threads)
        t.join();

    // 测试不同级别
    APP_LOG_TRACE("This is a trace message (only visible in Debug)");
    APP_LOG_DEBUG("Debug message");
    APP_LOG_WARN("Warning: something might be wrong");
    APP_LOG_ERROR("Error occurred");

    // 测试 backtrace 自动 dump
    APP_LOG_CRITICAL("Critical error! Backtrace will be dumped automatically.");

    // 程序退出时自动调用 shutdown，无需手动调用
    return 0;
}
```

## 7.5 线程安全

init() 和 shutdown() 使用互斥锁保护，可安全多线程调用，但建议在单线程启动阶段调用。

日志宏内部通过 LogManager::get() 获取 logger，该函数返回 shared_ptr，复制是线程安全的。

spdlog 的异步日志器本身是线程安全的。

## 7.6 平台特定行为

### 7.6.1 Windows Debug 模式

如果 consoleOut 为 true，会自动调用 AllocConsole() 分配一个新的控制台窗口，并重定向 stdout 和 stderr 到该控制台，设置控制台代码页为 UTF-8（防止中文乱码）。

可选择启用 VS 输出窗口（msvc_sink），已在代码中注释，需要可取消注释。

### 7.6.2 其他平台
如需在 Linux/macOS 上分配控制台，需修改相应代码（通常不需要，因为标准输出已存在）。

## 7.7 注意事项
* 日志级别控制：日志级别由编译选项自动决定，在 Debug 模式（_DEBUG 定义）：启用 TRACE 及以上所有级别，日志立即刷新。在 Release 模式（_RELEASE 定义）：启用 INFO 及以上级别，WARN 及以上立即刷新。可通过修改 SPDLOG_ACTIVE_LEVEL 宏来调整。
* 崩溃保护机制：默认缓存最近 32 条日志。当记录 CRITICAL 级别日志时，会自动 dump 缓存的 backtrace 到所有 sink。每 3 秒强制将缓冲区内容写入磁盘，以减少程序崩溃时的日志丢失风险；
* 宏定义顺序：必须在包含 spdlog 头文件之前定义 SPDLOG_ACTIVE_LEVEL，否则日志级别可能不正确；
* 日志文件权限：确保程序对日志目录有写入权限，否则初始化时会输出错误信息到 stderr；
* 异步线程池：默认队列大小为 8192，单个后台线程。若日志量极大（如每秒数万条），可调整 init_thread_pool 参数。当队列满时，策略为 block（阻塞主线程），保证日志不丢失。可改为 discard 以牺牲日志保性能；
* 多次初始化：多次调用 init 会关闭之前的日志器并重新创建，之前的日志文件仍保留；
* 静态析构顺序： Log Manager 的静态成员在程序退出时自动析构，会调用 shutdown() 。但如果其他静态对象的析构函数中仍使用日志宏，可能导致崩溃。建议在 main 函数返回前显式调用 shutdown() ，或确保日志宏在静态析构阶段不再使用；
* 获取原始 logger：若需要直接操作 spdlog logger，可通过 core::LogManager::get() 直接获取指针：

```cpp
// 假设我们想动态修改级别
auto logger = core::LogManager::get();
logger->set_level(spdlog::level::debug);
```

* 控制台颜色：控制台输出带有颜色，便于区分日志级别（stdout_color_sink）。若不需要颜色，可改用 stdout_sink；
* 与其他模块集成：日志管理器已在其他模块（Config Manager 、 Task Manager 、 Service Manager 等）中使用，通过 APP_LOG_XXX 宏输出日志，无需额外配置。
